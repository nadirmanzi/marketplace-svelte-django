## Architecture Overview

### Technology Stack
- **Framework**: Django 4.x with Django REST Framework
- **Authentication Library**: django-allauth (Headless mode)
- **Token System**: SimpleJWT (djangorestframework-simplejwt)
- **Token Strategy**: Custom `SimpleJWTTokenStrategy` integrating AllAuth headless with SimpleJWT
- **Adapter**: Custom `CustomHeadlessAdapter` for token rotation support

### Key Components
1. **AllAuth Headless**: Provides headless API endpoints for authentication
2. **Custom Token Strategy**: Bridges AllAuth with SimpleJWT token generation
3. **Custom Adapter**: Handles token rotation in API responses
4. **Authentication Service**: Business logic for user authentication
5. **Token Service**: Token lifecycle management (create, refresh, invalidate)
6. **Custom JWT Authentication**: Validates tokens with session versioning
7. **Password Expiration Middleware**: Enforces password expiration policy

### Authentication Backends
```python
AUTHENTICATION_BACKENDS = [
    "django.contrib.auth.backends.ModelBackend",  # Admin login
    "allauth.account.auth_backends.AuthenticationBackend",  # Email-based auth
]
```

## User Registration Flow

### Endpoint
`POST /users/auth/v1/app/signup/`

### Flow Diagram
```
Client Request
    ↓
AllAuth Headless SignupView
    ↓
Input Validation (email, password)
    ↓
User Creation (User.objects.create_user)
    ↓
Password Hashing (Django's make_password)
    ↓
User Saved to Database
    ↓
user_signed_up Signal Fired
    ↓
Email Verification Email Sent (if ACCOUNT_EMAIL_VERIFICATION = "mandatory")
    ↓
Response: JWT Tokens (if email verification not required)
    OR
Response: Email Verification Required
```

### Request Format
```json
{
  "email": "user@example.com",
  "password1": "SecurePassword123!",
  "password2": "SecurePassword123!"
}
```

### Response Format
**If email verification is mandatory:**
```json
{
  "status": "ok",
  "data": {
    "user": {
      "id": "uuid",
      "email": "user@example.com"
    }
  },
  "meta": {
    "message": "Verification email sent"
  }
}
```

**If email verification is optional:**
```json
{
  "status": "ok",
  "data": {
    "user": {
      "id": "uuid",
      "email": "user@example.com"
    }
  },
  "meta": {
    "access": "eyJ...",
    "refresh": "eyJ..."
  }
}
```

### Validation Rules
- Email must be unique (case-insensitive)
- Password must meet Django validators:
  - Minimum 8 characters (ACCOUNT_PASSWORD_MIN_LENGTH)
  - Not too common
  - Not entirely numeric
  - Not too similar to user attributes
- Email format validation

### Signal Handlers
- `user_signed_up`: Logs registration event for audit trail

### Configuration
```python
ACCOUNT_SIGNUP_FIELDS = ["email*", "password1"]
ACCOUNT_EMAIL_VERIFICATION = "mandatory"
ACCOUNT_AUTHENTICATION_METHOD = "email"
ACCOUNT_USERNAME_REQUIRED = False
```

## Login Flow

### Endpoint
`POST /users/auth/v1/app/login/`

### Flow Diagram
```
Client Request (email + password)
    ↓
AllAuth Headless LoginView
    ↓
AuthenticationService.authenticate_user()
    ↓
Django authenticate() with email/password
    ↓
Check: user.is_active
    ↓
Check: user.is_soft_deleted
    ↓
Check: user.email_verification_status (if mandatory)
    ↓
TokenService.create_login_tokens()
    ↓
CustomRefreshToken.for_user(user)
    ↓
CustomAccessToken created from refresh token
    ↓
Tokens stored in OutstandingToken table (if blacklist enabled)
    ↓
Optional: TokenService.cleanup_previous_tokens() (if SINGLE_ACTIVE_SESSION)
    ↓
AuthenticationService.set_last_login(user)
    ↓
Response: JWT Tokens in Authorization header + response body
```

### Request Format
```json
{
  "email": "user@example.com",
  "password": "SecurePassword123!"
}
```

### Response Format
**Headers:**
```
Authorization: Bearer eyJ0eXAiOiJKV1QiLCJhbGc...
```

**Body:**
```json
{
  "status": "ok",
  "data": {
    "user": {
      "id": "uuid",
      "email": "user@example.com"
    }
  },
  "meta": {
    "access": "eyJ...",
    "refresh": "eyJ..."
  }
}
```

### Token Claims
**Access Token:**
- `user_id`: UUID string
- `email`: User email
- `session_version`: Integer (for forced logout on password change)
- `exp`: Expiration timestamp (10 minutes)
- `iat`: Issued at timestamp
- `token_type`: "access"

**Refresh Token:**
- `user_id`: UUID string
- `email`: User email (cached for access token generation)
- `session_version`: Integer
- `exp`: Expiration timestamp (1 day)
- `iat`: Issued at timestamp
- `jti`: JWT ID (for blacklisting)

### Account Status Checks
1. **is_active**: Must be `True`
2. **is_soft_deleted**: Must be `False`
3. **email_verification_status**: Must be `"verified"` if `ACCOUNT_EMAIL_VERIFICATION = "mandatory"`

### Rate Limiting
- Throttled via `LoginRateThrottle` (100/min default)
- Configured in `REST_FRAMEWORK.DEFAULT_THROTTLE_RATES`

### Audit Logging
- Successful logins logged with user email
- Failed logins logged with error reason
- Token creation logged

## Token Management

### Token Types

#### 1. Access Token
- **Lifetime**: 10 minutes (`SIMPLE_JWT.ACCESS_TOKEN_LIFETIME`)
- **Purpose**: Authenticate API requests
- **Storage**: Client-side (memory/localStorage)
- **Delivery**: Authorization header (`Bearer <token>`)
- **Claims**: Minimal (user_id, email, session_version)

#### 2. Refresh Token
- **Lifetime**: 1 day (`SIMPLE_JWT.REFRESH_TOKEN_LIFETIME`)
- **Purpose**: Obtain new access tokens
- **Storage**: Client-side (secure storage recommended)
- **Delivery**: Request body or header
- **Claims**: user_id, email, session_version, jti

### Token Generation
```python
# Custom token classes extend SimpleJWT
refresh = CustomRefreshToken.for_user(user)
access = refresh.access_token  # Zero DB lookup (email cached in refresh)
```

### Token Storage
- **OutstandingToken**: Tracks all issued refresh tokens (if blacklist enabled)
- **BlacklistedToken**: Tracks invalidated tokens (if blacklist enabled)

### Token Validation
```python
# CustomJWTAuthentication validates:
1. Token signature
2. Token expiration
3. Token blacklist status
4. Session version match (user.session_version == token.session_version)
```

## Token Refresh Flow

### Endpoint
`POST /users/auth/v1/app/token/refresh/`

### Flow Diagram
```
Client Request (refresh token)
    ↓
AllAuth Headless Token Refresh Endpoint
    ↓
SimpleJWTTokenStrategy.refresh_access_token()
    ↓
TokenService.refresh_token(refresh_token)
    ↓
Validate Refresh Token (signature, expiration)
    ↓
Check Blacklist Status
    ↓
Extract Access Token from Refresh Token (zero DB lookup)
    ↓
If ROTATE_REFRESH_TOKENS = True:
    - Blacklist old refresh token
    - Create new refresh token
    - Store new refresh token in request._allauth_rotated_refresh_token
    ↓
Return Access Token
    ↓
CustomHeadlessAdapter.respond_to_token_refresh()
    ↓
Read rotated refresh token from request
    ↓
Include refresh token in response body (if rotation occurred)
    ↓
Response: New access token (header) + new refresh token (body, if rotated)
```

### Request Format
```json
{
  "refresh": "eyJ0eXAiOiJKV1QiLCJhbGc..."
}
```

### Response Format
**With Token Rotation:**
```json
{
  "status": "ok",
  "meta": {
    "access": "eyJ...",
    "refresh": "eyJ..."  // New refresh token
  }
}
```

**Without Token Rotation:**
```json
{
  "status": "ok",
  "meta": {
    "access": "eyJ..."
    // Same refresh token continues to work
  }
}
```

### Token Rotation
- **Enabled**: `ROTATE_REFRESH_TOKENS = True`
- **Behavior**: Old refresh token blacklisted, new one issued
- **Security**: Prevents token reuse attacks
- **Client Action**: Must update stored refresh token when rotation occurs

### Zero-DB Token Refresh
- Email cached in refresh token payload
- Access token generated from refresh payload (no user lookup)
- Critical for performance with token rotation

**Flow:**
```

    ↓

    ↓
```

- Token invalidated if password changed since issue (session version check)

## Password Change Flow

### Endpoint
`POST /users/auth/v1/app/password/change/`

### Flow Diagram
```
Authenticated Request (current_password + new_password)
    ↓
AllAuth Headless Password Change View
    ↓
Verify Current Password
    ↓
Validate New Password (Django validators)
    ↓
User.set_password(new_password)
    ↓
password_changed_at = timezone.now()
    ↓
session_version += 1 (invalidates all existing tokens)
    ↓
password_changed Signal Fired
    ↓
Response: Success
```

### Request Format
```json
{
  "oldpassword": "CurrentPassword123!",
  "password1": "NewSecurePassword123!",
  "password2": "NewSecurePassword123!"
}
```

### Security Impact
- **All existing tokens invalidated**: Session version increment forces re-login
- **Password expiration reset**: `password_changed_at` updated
- **Audit logging**: Password change event logged

## Logout Flow

### Endpoint
`POST /users/auth/v1/app/logout/`

### Flow Diagram
```
Client Request (refresh token)
    ↓
AllAuth Headless Logout View
    ↓
TokenService.invalidate_token(refresh_token)
    ↓
Find OutstandingToken by jti
    ↓
Create BlacklistedToken entry
    ↓
Response: Success
```

### Request Format
```json
{
  "refresh": "eyJ0eXAiOiJKV1QiLCJhbGc..."
}
```

### Response Format
```json
{
  "status": "ok",
  "meta": {
    "message": "Logged out successfully"
  }
}
```

### Behavior
- Refresh token blacklisted (cannot be used for refresh)
- Access token remains valid until expiration (10 min)
- Client should discard both tokens

### Alternative: Token Invalidation
**Endpoint**: `POST /users/account/token_invalidate/` (custom view)

- Requires valid access token
- Useful for revoking specific device sessions
- Same blacklist mechanism

## Session Management

### Session Versioning
- **Purpose**: Force logout on password change or security events
- **Implementation**: `user.session_version` integer field
- **Increment Triggers**:
  - Password change (`User.set_password()`)
  - Manual security action (admin)

### Token Validation
```python
# CustomJWTAuthentication.get_user()
token_version = validated_token.get("session_version", 0)
user_version = user.session_version

if token_version != user_version:
    raise AuthenticationFailed("Session invalid")
```

### Single Active Session
- **Setting**: `SINGLE_ACTIVE_SESSION = False` (currently disabled)
- **Behavior**: When enabled, blacklists all previous tokens on login
- **Implementation**: `TokenService.cleanup_previous_tokens()`

### Password Expiration
- **Default**: 90 days (`password_expires_in_days`)
- **Enforcement**: `PasswordExpirationMiddleware`
- **Behavior**: Blocks API access, allows password change endpoint
- **Exempt URLs**: Login, logout, token refresh, password change

### Middleware Flow
```
Request
    ↓
PasswordExpirationMiddleware.process_view()
    ↓
Check: user.password_expired
    ↓
If expired AND not exempt URL:
    Return 403: "password_expired"
    ↓
Otherwise: Continue to view
```

## Security Features

### 1. Token Security
- **Short-lived access tokens**: 10 minutes
- **Token rotation**: Prevents reuse attacks
- **Blacklisting**: Server-side token invalidation
- **Minimal claims**: Only user_id, email, session_version

### 2. Password Security
- **Hashing**: Django's PBKDF2 with SHA256
- **Validators**: Length, complexity, common password checks
- **Expiration**: 90-day policy with middleware enforcement
- **Session invalidation**: All tokens invalidated on password change

### 3. Email Verification
- **Mandatory**: Required before account activation
- **Token expiration**: 5 hours
- **Status tracking**: `email_verification_status` field

### 4. Account Protection
- **Soft delete**: Accounts can be marked deleted (preserves data)
- **Deactivation**: Accounts can be temporarily disabled
- **Status checks**: Enforced on login

### 5. Rate Limiting

### Common Error Responses

#### Authentication Errors

```json
  }
}
```

```json

  "status": "error",
```

### HTTP Status Codes
- `200 OK`: Success
- `201 Created`: Resource created (signup)
- `400 Bad Request`: Validation error
- `401 Unauthorized`: Authentication required / Invalid token
- `403 Forbidden`: Account deactivated / Password expired
- `404 Not Found`: Resource not found
- `409 Conflict`: Resource conflict (e.g., already logged in)
- `429 Too Many Requests`: Rate limit exceeded

## Best Practices

### Client Implementation
1. **Token Storage**: Store refresh token securely (not in localStorage for sensitive apps)
2. **Token Refresh**: Refresh access token before expiration (e.g., at 9 minutes)
3. **Token Rotation**: Always check for new refresh token in response
4. **Error Handling**: Handle 401 by attempting token refresh, then re-login
5. **Logout**: Call logout endpoint to blacklist refresh token

### Security Recommendations
1. **HTTPS**: Always use HTTPS in production
2. **Token Storage**: Use secure storage mechanisms (httpOnly cookies if possible)
3. **Token Rotation**: Keep enabled for security
4. **Password Policy**: Enforce strong password requirements
5. **Rate Limiting**: Monitor and adjust rate limits based on usage
6. **Audit Logs**: Monitor authentication events for suspicious activity

## Troubleshooting

### Common Issues

#### Token Refresh Returns 401
- **Cause**: Refresh token expired or blacklisted
- **Solution**: User must re-login

#### Session Invalid After Password Change
- **Expected**: All tokens invalidated on password change
- **Solution**: User must re-login

#### Email Verification Not Working
- **Check**: `ACCOUNT_EMAIL_VERIFICATION` setting
- **Check**: Email backend configuration
- **Check**: Frontend URL configuration

#### Password Expired Blocking Access
- **Expected**: Middleware enforces password expiration
- **Solution**: User must change password via password change endpoint
