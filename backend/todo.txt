## [FIXED] Bug 1 — Expired Password Login Bumped session_version Unnecessarily

### Root Cause
In `users/views/auth_views.py`, the `session_version` increment and token generation
happened *before* checking if the user's password was expired. This caused a spurious
DB write (bumping `session_version`) even for users who would be immediately blocked,
and tokens were generated (then discarded or included in the 403 response) needlessly.

### Fix Applied
Moved the `password_expired` check to immediately after `serializer.is_valid()`,
*before* `session_version += 1` and token generation. The 403 early-return now has
zero side effects on the DB or session state.

```python
# BEFORE (wrong order):
user.session_version += 1        # ← bumped before expiry check
user.save(...)
...
if user.password_expired:
    return Response(..., status=403)  # side effects already happened

# AFTER (correct):
if user.password_expired:
    return Response(..., status=403)  # early exit, no side effects
user.session_version += 1
user.save(...)
```

**File changed:** `users/views/auth_views.py` (lines 91–136)

## [FIXED] Bug 2 — Required Category Attributes Not Enforced on Product Create

### Root Cause
In `catalog/services/product_services.py`, the guard condition was:
```python
if attributes or category_instance:
    AttributeValueService.validate_and_assign(product, attributes, is_variant=False)
```
When a client sends no `attributes` key (serializer defaults to `{}`), `attributes`
is falsy `{}`. If `category_instance` is truthy, `validate_and_assign` would be
called — but `attributes` (not `attributes or {}`) was passed, so an empty dict
was correct. The deeper bug: when a category IS set but NO attributes are sent, the
required-attribute enforcement loop in `validate_and_assign` would correctly raise —
but only if the outer `if` passes. The `attributes or {}` fallback was missing,
meaning if somehow `attributes` ended up as `None`, the service would break silently.

### Fix Applied
Changed the guard to always call `validate_and_assign` when a category is present,
and added an `attributes or {}` fallback for safety:

```python
# BEFORE:
if attributes or category_instance:
    AttributeValueService.validate_and_assign(product, attributes, is_variant=False)

# AFTER:
if category_instance:
    AttributeValueService.validate_and_assign(product, attributes or {}, is_variant=False)
```

**File changed:** `catalog/services/product_services.py` (line 84)

## [FIXED] Bug 3 — Middleware 403 Response Shape Inconsistent with DRF Errors

### Root Cause
`PasswordExpirationMiddleware.process_view()` returned a `JsonResponse` with:
```json
{"error": "password_expired", "detail": "...", "change_password_url": "..."}
```
DRF's custom exception handler produces `{"detail": "...", "code": "..."}`.
The extra `"error"` key made the middleware response shape differ from all other
API errors, requiring frontend clients to handle two different error shapes.

### Fix Applied
Normalized the response to DRF convention — removed `"error"` key, added `"code"`:
```json
{"detail": "...", "code": "password_expired", "change_password_url": "..."}
```

**File changed:** `users/middleware.py` (lines 73–80)

## [FIXED] Bug 4 — Middleware Authentication Crash: 'WSGIRequest' has no attribute '_request'

### Root Cause
`_authenticate_request()` passed the raw Django `WSGIRequest` directly to each DRF
authenticator's `.authenticate()` method. DRF authenticators (including
`JWTAuthentication` and its parent `BaseAuthentication`) internally access
`request._request` — an attribute that only exists on DRF's own `Request` wrapper,
not on the bare `WSGIRequest`. This caused:

```
Unexpected error during middleware authentication: 'WSGIRequest' object has no attribute '_request'
```

This exception was caught by the broad `except Exception`, logged as a warning, and
execution continued — but authentication always failed silently, meaning
`process_view` never blocked on expired passwords for JWT-authenticated routes.

### Fix Applied
Wrapped the raw `WSGIRequest` in a `rest_framework.request.Request` object (with the
collected authenticator instances) before calling `.authenticate()`. This gives DRF
authenticators the `._request` they need, plus the full DRF request interface:

```python
from rest_framework.request import Request as DRFRequest

authenticators = [...]           # instantiated from DEFAULT_AUTHENTICATION_CLASSES
drf_request = DRFRequest(request, authenticators=authenticators)

for authenticator in authenticators:
    auth_result = authenticator.authenticate(drf_request)
```

The instantiation loop was also refactored to resolve string class paths once upfront
and reuse instances for both the DRFRequest wrapper and the authenticate loop.

**File changed:** `users/middleware.py` (lines 84–128)
